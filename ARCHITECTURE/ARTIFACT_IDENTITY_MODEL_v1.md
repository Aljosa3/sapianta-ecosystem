# SAPIANTA Artifact Identity Model v1

## Document Role

This document defines deterministic governed artifact identity semantics for the SAPIANTA ecosystem.

It is documentation-only. It does not implement hashing, runtime acceptance, replay tooling, promotion automation, or activation behavior.

Authoritative foundation references:
- `ARCHITECTURE/REPLAY_LINEAGE_MODEL_v1.md`
- `ARCHITECTURE/REPOSITORY_INTERACTION_FLOW_v1.md`
- `ARCHITECTURE/RUNTIME_ACCEPTANCE_GATE_v1.md`

## Canonical Artifact Identity

A governed artifact identity is the deterministic reference used to identify an artifact across proposal, validation, replay, audit, review, approval, and promotion contexts.

Canonical identity may include:
- artifact name
- artifact type
- artifact path
- source repository
- source branch
- source commit
- deterministic hash
- source lineage identity
- validation lineage
- replay lineage
- governance lineage
- approval lineage

## Artifact Hash Identity

Artifact hash identity is the deterministic content identity of an artifact at a specific point in lineage.

Hash identity must be stable for the same artifact content and context.

This document does not define a hashing implementation.

## Repository Attribution

Every governed artifact should preserve repository attribution:
- originating repository
- originating branch
- originating commit when available
- originating workspace authority
- proposal source when generated by factory

Repository attribution must remain visible across promotion and acceptance review.

## Source Lineage Identity

Source lineage identity links an artifact to its origin and movement history.

It may include:
- domain concept reference
- factory proposal reference
- validation evidence reference
- replay evidence reference
- promotion review reference
- governance approval reference

## Deterministic Replay Identity

Deterministic replay identity links an artifact to replay inputs, replay environment assumptions, replay outputs, and reproducibility evidence.

Artifact identity MUST remain stable across replay validation.

If replay validation produces a different artifact identity, the artifact must be treated as changed and must not silently inherit prior approval.

## Immutable Lineage References

Lineage references must not be rewritten to hide origin, validation evidence, review evidence, rejection, rollback, or revocation.

Canonical state documents may reference current identity status, but lineage documents must preserve historical identity continuity.

## Lineage Integrity Failure Rule

- Silent lineage truncation is prohibited.
- Lineage loss is a governance integrity failure.
- Missing lineage blocks governed promotion.
- Lineage must not be silently inferred.
- Lineage recovery requires explicit governance review.

## Authority Constraints

Artifact identity does not imply:
- execution authority
- runtime activation
- governance activation
- production approval
- factory promotion authority

Identity is evidence and lineage. It is not authority by itself.
