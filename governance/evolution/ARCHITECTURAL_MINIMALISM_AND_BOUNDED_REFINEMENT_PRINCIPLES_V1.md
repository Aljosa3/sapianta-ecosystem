# 2. ARCHITECTURAL MINIMALISM PRINCIPLES

SAPIANTA/AiGOL architectural evolution MUST prefer:

- refinement before expansion;
- semantic correction before redesign;
- stabilization before capability multiplication;
- bounded systems before orchestration inflation;
- deterministic simplicity before adaptive complexity;
- replay-safe continuity before new subsystem introduction;
- governance-visible justification before complexity increase.

The following equivalences are explicitly rejected:

- feature count != stability;
- subsystem count != capability maturity;
- autonomy != correctness;
- complexity != intelligence;
- orchestration volume != governance maturity;
- abstraction depth != operational trust.

---

# 3. BOUNDED REFINEMENT DISCIPLINE

When operational issues appear, the default response MUST be bounded
refinement:

- prefer semantic refinement;
- prefer bounded correction;
- prefer deterministic hardening;
- prefer continuity preservation;
- prefer replay-visible evidence;
- prefer fail-closed clarification.

Operational semantic discoveries are NOT justification for architecture
explosion.

An operational defect, ambiguity, or continuity weakness MUST first be
evaluated as a candidate for refinement inside the existing substrate.

---

# 4. ARCHITECTURE EXPANSION CRITERIA

Architectural expansion MAY become justified only when all of the
following are true:

- refinement insufficiency is demonstrated;
- deterministic replay evidence exists;
- rollback strategy is bounded and documented;
- constitutional compatibility is preserved;
- operational stability is demonstrated;
- governance review is required;
- complexity increase has bounded purpose.

Expansion MUST justify:

- why refinement is insufficient;
- why the existing substrate cannot support the requirement;
- why complexity increase is necessary;
- why replay and governance impact is acceptable;
- how rollback and lineage visibility remain preserved.
