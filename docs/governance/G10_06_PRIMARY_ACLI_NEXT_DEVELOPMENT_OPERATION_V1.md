Status: primary ACLI Next development operation established.

Final verdict: PRIMARY_ACLI_NEXT_DEVELOPMENT_OPERATION_ESTABLISHED

## 1. Executive Summary

Generation 10 now formally establishes ACLI Next as the primary operational interface for day-to-day governed Platform Core development.

This operating policy follows certified findings:

- Generation 9 certified AiGOL as the governed development foundation.
- G10-01 canonicalized the hybrid-primary operating model.
- G10-04 implemented ACLI Next daily operational exposure.
- G10-04A confirmed the exposure architecture.
- G10-05 determined that ACLI Next is ready as the primary development interface.

Operating rule:

```text
All certified governed Platform Core development shall begin in ACLI Next by default.
```

External tools remain available only as bounded operational exceptions where certified ACLI Next / Platform Core coverage ends.

## 2. Canonical Operating Policy

From this point forward, the canonical day-to-day development path is:

```text
Human
-> ACLI Next
-> Governed Development Workflow
-> Platform Core
-> OCS
-> Governance
-> Worker Platform
-> Replay
-> Platform Digital Twin
-> Architectural Health
-> Human review
-> Architecture review
-> Certification
```

This policy does not redesign Platform Core.

It operationalizes the certified architecture already established across Generations 6 through 10.

## 3. Default Workflow

Every development task shall begin by determining whether it can enter ACLI Next.

Default flow:

1. Human states the development intent in ACLI Next.
2. ACLI Next captures and presents the governed workflow context.
3. Platform Core coordinates the Governed Development Workflow.
4. OCS forms candidates or proposals where required.
5. Governance manages approval, authorization, admissibility, and certification.
6. Worker Platform executes bounded authorized operations where required.
7. Replay records evidence and supports reconstruction.
8. Platform Digital Twin projects canonical architectural evidence.
9. Architectural Health produces advisory findings where applicable.
10. Human reviews results, evidence, findings, and final verdicts.

ACLI Next remains:

```text
show
guide
delegate
```

It does not authorize, execute, certify, own Replay, or perform Architectural Health reasoning.

## 4. Mandatory ACLI Next Entry

The following work shall normally begin in ACLI Next:

- readiness reviews;
- specifications;
- implementation tasks;
- architecture reviews;
- certification reviews;
- governed artifact creation;
- governed mutation planning;
- validation and validation suite review;
- rollback review and execution requests where certified;
- local governed development status review;
- Architectural Health review;
- Replay evidence review;
- primary development dashboard review.

If certified Platform Core capability exists, external manual operation is not the default path.

## 5. External Exceptions

External tools remain limited to bounded operational exceptions.

Allowed external exceptions:

| Exception | Reason |
| --- | --- |
| Git remote workflows | Remote push, fetch, pull, branch, merge, rebase, and pull request operations remain outside certified scope. |
| Dependency management | Package manager operations require package, registry, lockfile, network, and validation policy. |
| Deployment | Deployment crosses release, environment authority, runtime activation, rollback, and operational safety boundaries. |
| Exceptional environment operations | Local OS, infrastructure, credential, or environment actions may fall outside Worker Platform certification. |
| Unsupported investigations | Deep debugging or emergency analysis may require Codex or terminal when no certified workflow exists. |

External exceptions must remain visible.

They must not become:

- alternative development workflows;
- new authority layers;
- Governance bypasses;
- Replay bypasses;
- Worker Platform bypasses;
- uncontrolled deployment paths.

## 6. Daily Operating Principles

### 6.1 Begin In ACLI Next

Every development task begins in ACLI Next unless the human already knows the task is an uncertified operational exception.

Even when an exception is expected, ACLI Next should be used to record why the exception exists when practical.

### 6.2 Remain Inside Certified Coverage

Development remains inside ACLI Next and the Governed Development Workflow when:

- the requested action maps to certified capability;
- Governance can authorize it;
- Worker Platform can execute it;
- Replay can record evidence;
- validation can run through certified validation capability;
- Architectural Health can observe deterministic evidence.

### 6.3 Preserve Replay Continuity

Replay continuity must be preserved for all governed activity.

When hybrid execution is required, the operator must record:

- why external execution was required;
- which external tool was used;
- what operation was performed;
- what scope was affected;
- what validation followed;
- what evidence remains outside Replay;
- when control returned to ACLI Next.

### 6.4 Preserve Governance Continuity

Governance continuity must be preserved by ensuring:

- external tools never authorize themselves;
- manual execution is not retroactively treated as governed execution;
- unsupported boundaries remain visible;
- certification artifacts disclose hybrid exceptions;
- future capability closure follows `Reuse -> Canonicalization -> Extension`.

### 6.5 Preserve Architectural Health Visibility

Architectural Health visibility must be preserved by:

- surfacing advisory findings in ACLI Next;
- keeping findings advisory-only;
- requiring human and Governance review for significant boundary concerns;
- not allowing Architectural Health to repair, reject, approve, or certify.

### 6.6 Return To ACLI Next

After any hybrid exception, development returns to ACLI Next as soon as certified workflow coverage resumes.

The hybrid path is a temporary detour, not a second operating model.
