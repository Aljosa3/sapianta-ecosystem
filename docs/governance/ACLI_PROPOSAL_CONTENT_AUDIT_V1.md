## 1. Purpose

This artifact audits the governed development proposal generation path with a focus on proposal content, proposal naming, and operator understanding.

The audit determines:

- why requested artifact names are not preserved;
- why generated hash-based artifact names are shown instead;
- whether operator intent can be reflected directly in proposal summaries;
- whether proposal previews can show intended repository changes before approval.

This audit does not modify governance behavior, approval behavior, workflow behavior, repository mutation behavior, validation, or replay.

## 3. Current Proposal Generation Flow

The current governed development bridge receives:

- the human prompt;
- routing decision artifacts;
- workflow selection artifacts;
- universal intake artifacts;
- workspace root;
- replay directory.

It then computes a deterministic seed:

```text
seed = replay_hash(
    {
        "prompt_id": prompt_id,
        "human_prompt": human_prompt,
        "workflow_selection_hash": workflow_selection["artifact_hash"],
    }
)[7:23]
```

That seed is then passed into:

```text
_governance_artifact(seed=seed, human_prompt=human_prompt)
_repository_file_mutation(seed=seed, human_prompt=human_prompt)
```

The generated governance artifact title is:

```text
ACLI_GOVERNED_DEVELOPMENT_<SEED>_V1
```

The generated governance artifact path is:

```text
docs/governance/ACLI_GOVERNED_DEVELOPMENT_<SEED>_V1.md
```

The generated runtime marker file path is:

```text
aigol/runtime/acli_governed_development_<seed>.py
```

## 4. Why Requested Artifact Names Are Not Preserved

Requested artifact names are not preserved because the proposal generation path does not extract artifact identifiers from the human prompt.

Example prompt:

```text
Create governance artifact ACLI_USAGE_GUIDELINES_V1 documenting recommended operator practices for using ACLI as the primary development interface.
```

The current runtime preserves this prompt in:

- the request artifact;
- the generated governance artifact content under `Human request`;
- the generated runtime marker constant value.

However, the runtime does not parse `ACLI_USAGE_GUIDELINES_V1` into a structured field such as:

```text
requested_artifact_name
requested_artifact_title
requested_target_path
```

Because no structured requested name exists, `_governance_artifact(...)` cannot use it. It always derives the artifact title from the replay hash seed.

## 5. Why Hash-Based Artifact Names Are Shown

Hash-based artifact names are shown because `_governance_artifact(...)` currently constructs names from the deterministic replay seed:

```text
title = f"ACLI_GOVERNED_DEVELOPMENT_{seed.upper()}_V1"
```

This gives the runtime several useful properties:

- deterministic generated names;
- collision resistance across prompts;
- replay-stable proposal artifacts;
- no dependency on ambiguous natural-language parsing;
- safe fallback when no artifact name is requested.

The cost is operator comprehension:

- the generated name does not match the operator's requested artifact name;
- proposal previews show `ACLI_GOVERNED_DEVELOPMENT_<hash>_V1` instead of the intended artifact;
- the operator cannot easily verify that ACLI understood the named artifact target.

## 6. Proposal Content Findings

Current governance artifact content is generic:

```text
# ACLI_GOVERNED_DEVELOPMENT_<SEED>_V1

Status: Generated Proposal

Purpose: Replay-visible governance artifact generated by the ACLI governed development bridge.

Human request:

<original prompt>

Boundaries:

- Human approval remains required.
- Repository mutation remains worker-governed.
- Replay remains source of truth.
```

Findings:

- The original human request is preserved.
- The content is replay-safe.
- The content is generic rather than tailored to the requested artifact.
- The purpose line does not reflect the operator's stated purpose.
- The artifact title does not reflect the requested artifact name.
- The proposal summary does not preview the proposed file content.

## 7. Proposal Naming Findings

Current naming source:

```text
replay hash seed
```

Current name shape:

```text
ACLI_GOVERNED_DEVELOPMENT_<SEED>_V1
```

Operator-expected name shape:

```text
<REQUESTED_ARTIFACT_NAME>.md
```

Example:

```text
docs/governance/ACLI_USAGE_GUIDELINES_V1.md
```

The current runtime therefore prioritizes deterministic uniqueness over operator-intent preservation.

This is safe, but not human-friendly.

## 8. Operator Understanding Impact

The current proposal can confuse an operator because the visible target path appears unrelated to the prompt.

Example:

Prompt:

```text
Create governance artifact ACLI_USAGE_GUIDELINES_V1 documenting recommended operator practices...
```

Proposal target path:

```text
docs/governance/ACLI_GOVERNED_DEVELOPMENT_E5BAAA6109EE2B3D_V1.md
```

Likely operator interpretation:

- "Why did ACLI choose a different artifact name?"
- "Did ACLI understand the requested name?"
- "Will the requested artifact actually be created?"
- "What content will be written if I approve?"

This is a proposal clarity issue, not a governance authority issue.

## 9. Can Operator Intent Be Reflected Directly?

Yes, operator intent can be reflected directly in proposal summaries without changing governance behavior.

Minimum non-behavioral improvement:

- display the extracted or detected requested name if available;
- display the original prompt purpose;
- distinguish requested target from generated target;
- state whether the target path is inferred or generated.

Example operator summary:

```text
Requested artifact name detected:
ACLI_USAGE_GUIDELINES_V1

Current generated target:
docs/governance/ACLI_GOVERNED_DEVELOPMENT_E5BAAA6109EE2B3D_V1.md

Note:
The current implementation records your requested name in the proposal content, but does not yet use it as the target filename.
```

This can be done as presentation only if no mutation target changes.

Using the requested name as the actual target path would be behavior change and should require implementation review and tests.

## 11. Root Cause Classification

| Issue | Root cause | Classification |
| --- | --- | --- |
| Requested names not preserved as filenames | No artifact-name extraction from prompt | Missing proposal intent extraction |
| Hash-based names shown | Seed-based deterministic fallback is always used | Intent-insensitive naming |
| Proposal summary lacks requested name | Renderer uses proposal target paths only | Presentation gap |
| Proposed content not previewed | Renderer omits `proposed_content` summary | Presentation gap |
| Operator cannot compare requested vs generated target | No structured requested target field | Evidence visibility gap |

## 12. Governance Boundary Assessment

The current behavior preserves governance boundaries:

- no mutation occurs before approval;
- proposal hash binding remains intact;
- repository mutation remains worker-governed;
- validation remains allowlisted;
- replay records the proposal;
- human approval remains required.

Improving proposal summaries is safe if it only renders existing proposal data.

Changing target paths to use requested artifact names is a runtime behavior change and must preserve:

- deterministic sanitization;
- allowed directory boundaries;
- collision handling;
- proposal hash binding;
- approval before mutation;
- replay of extracted-name evidence.

## 13. Implementation Recommendations

### P0: Proposal Preview Rendering

Add a pre-approval proposal preview to the operator summary.

Include:

- proposed governance artifact path;
- proposed title;
- proposed purpose;
- content section summary;
- repository mutation paths;
- mutation operations;
- validation command.

This is presentation-only and does not change governance behavior.

### P0: Requested Name Visibility

Display the original human-requested artifact name when it can be detected safely.

If no structured detection exists yet, state:

```text
Requested artifact name: not extracted by current runtime
```

This prevents the operator from mistaking the generated hash name for intent preservation.

### P1: Deterministic Artifact Name Extraction

Add deterministic extraction for explicit artifact identifiers such as:

```text
ACLI_USAGE_GUIDELINES_V1
TEST_ACLI_BRIDGE_V1
```

Extraction must be:

- deterministic;
- bounded to safe identifier patterns;
- replay-visible;
- non-authoritative until used in proposal generation;
- fail-closed on ambiguous multiple identifiers.

### P1: Requested Target Path Proposal

After deterministic extraction exists, proposal generation may use:

```text
docs/governance/<REQUESTED_ARTIFACT_NAME>.md
```

only when:

- the identifier passes validation;
- the target directory is allowed;
- no unsafe path components exist;
- collision behavior is defined;
- replay records the extracted name and target decision.

### P1: Purpose Extraction

Extract a plain-language purpose from prompts using deterministic patterns such as:

```text
documenting ...
defining ...
recording ...
explaining ...
```

Fallback must remain the current generic purpose.

### P2: Proposal Diff Preview

Render a small preview or synthetic diff before approval.

This should show enough for operator understanding without executing mutation.

## 14. Replay Impact

Any future implementation should replay:

- original human prompt;
- extracted artifact name, if any;
- extraction confidence or deterministic match rule;
- selected target path;
- whether target path was generated or requested;
- proposed content preview hash;
- proposal summary rendered to the operator.

Replay must make it clear whether a name was:

- requested by the operator;
- extracted by ACLI;
- generated from hash fallback;
- changed by approval-time mutation logic.

## 15. Final Verdict

The governed development proposal generation path is safe and replay-visible, but it currently favors deterministic hash-generated names over operator-intent-preserving names.

Requested artifact names are not preserved because no deterministic artifact-name extraction feeds proposal generation.

Proposal previews can be improved immediately by rendering existing proposal content before approval. Using requested artifact names as actual target paths should be handled as a later bounded implementation with deterministic extraction, replay evidence, and tests.
