# SAPIANTA Explosion Domain Boundaries

## Boundary Statement

Explosion is not currently a formal runtime domain. It is a reconstructed latent acceleration capability.

Explosion must not be treated as autonomous execution authority.

## Allowed Current Scope

- documentation-only architectural memory
- inspection of acceleration-related artifacts
- research/evolution concept mapping
- sandbox governance review
- CAL/CCS/governed development state reconstruction
- dormant activation dependency tracking

## Forbidden Current Scope

- runtime activation
- autonomous execution
- autonomous git execution
- autonomous governance mutation
- autonomous Decision Spine mutation
- autonomous policy engine mutation
- production deployment
- self-modifying governance
- self-increasing autonomy level
- bypassing sandbox and promotion gates

## Related But Separate Domains

Explosion is not the same as:
- CAL
- CCS
- GAD
- ASF
- Research Runtime
- Strategy Evolution Engine
- Sandbox Model B
- Architecture Guardian

Those are adjacent systems or concepts. Explosion may later be defined as an umbrella acceleration domain, but that definition does not exist yet.

## Dangerous Activation Areas

- development autonomous loop as continuous execution
- CAL-generated task insertion
- auto-fix and repair orchestration
- research/evolution strategy mutation
- production promotion
- artifact registration with write behavior
- any path that writes to runtime, governance, policy, or production without human approval

## Boundary Preservation Rule

Any future Explosion work must first define the domain. No adjacent acceleration component may be implicitly promoted into Explosion authority.

Activation requires explicit ADR, explicit domain contract, milestone summary, SYSTEM_STATE update, CURRENT_FOCUS update, ROADMAP update, replay-safety review, and human approval.
